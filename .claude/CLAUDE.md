## Project Overview

Wealthfolio - Desktop investment tracker with local-first data. React + Vite
frontend, Tauri/Rust backend, SQLite storage. Also runs as a web app (Axum
server).

## Quick Commands

- Dev desktop: `pnpm tauri dev`
- Dev web: `pnpm run dev:web` (starts Axum + Vite, UI at localhost:1420)
- Dev addons: `VITE_ENABLE_ADDON_DEV_MODE=true pnpm tauri dev`
- Frontend tests: `pnpm test` (Vitest, watch mode)
- Single frontend test: `pnpm test -- path/to/file.test.ts`
- Rust tests: `cargo test`
- Single Rust test: `cargo test -p wealthfolio-core test_name`
- Type check: `pnpm type-check` (builds types first, then checks all packages)
- Lint: `pnpm lint`
- Format: `pnpm format`
- Full check: `pnpm check` (format + lint + type-check)
- E2E tests: `pnpm test:e2e` (Playwright)
- Clippy: `cargo clippy --workspace` (all warnings enabled in workspace config)

## Architecture

### Dual-Target Frontend (Compile-Time Adapter Pattern)

The frontend runs on both Desktop (Tauri IPC) and Web (HTTP REST) with
**zero runtime branching**. Vite resolves `@/adapters` to either
`src/adapters/tauri/` or `src/adapters/web/` based on `BUILD_TARGET` env var
(defaults to "tauri"). The `#platform` alias similarly resolves platform-specific
core modules (invoke, logger, isDesktop).

Both adapter directories export the same API surface. Shared adapters live in
`src/adapters/shared/` and use `invoke()` which resolves to Tauri IPC or HTTP
depending on build target.

### Rust Crate Dependency Flow

```
apps/tauri (Tauri IPC commands)     apps/server (Axum HTTP handlers)
         \                            /
          ↓                          ↓
    crates/core        — domain logic, service traits, repository traits
    crates/storage-sqlite — Diesel ORM, implements repository traits from core
    crates/market-data — quote fetching and caching
    crates/connect     — broker sync, external integrations
    crates/device-sync — device pairing, sync engine
    crates/ai          — AI chat with pluggable environment trait
```

Key separation: `core` defines traits, `storage-sqlite` implements them with
Diesel. Only `storage-sqlite` depends on Diesel.

### Request Flow (Example: Create Account)

1. Frontend adapter calls `invoke("create_account", { account })`
2. Tauri: `apps/tauri/src/commands/account.rs` / Server:
   `apps/server/src/api/account.rs`
3. Service layer: `crates/core/src/accounts/accounts_service.rs`
4. Repository: `crates/storage-sqlite/src/accounts/repository.rs` (Diesel, via
   `WriteHandle` for async transactions)
5. Domain events emitted (e.g., triggers portfolio recalc)

### Database (Diesel + SQLite)

- Schema: `crates/storage-sqlite/src/schema.rs` (auto-generated by Diesel CLI)
- Migrations: `crates/storage-sqlite/migrations/` (format:
  `YYYY-MM-DD-HHmmss_name`)
- Connection pooling: r2d2. Writes go through `WriteHandle::exec_tx()` for
  transactional execution
- Config: `crates/storage-sqlite/diesel.toml`

### Frontend Routing

Uses React Router (not TanStack Router). Routes defined in
`apps/frontend/src/routes.tsx`. Dynamic addon routes loaded at runtime.

### Key Frontend Libraries

- TanStack Query for data fetching/caching
- Tailwind CSS + Radix UI/Shadcn for styling
- Zod for validation
- Recharts for charts

## Code Conventions

### TypeScript / Frontend

- Functional components only, no classes
- Prefer interfaces over types; avoid enums (use maps)
- Named exports for components
- Lowercase-with-dashes for directories (e.g., `components/auth-wizard`)
- File structure: exported component → subcomponents → helpers → static → types
- Use `function` keyword for pure functions

### Rust

- snake_case for variables/functions, PascalCase for types/structs
- `thiserror` for custom error types, `?` operator for propagation
- `Result`/`Option` for error handling
- `tokio::test` for async tests
- `unsafe_code = "forbid"` workspace-wide

## Behavioral Guidelines

**Tradeoff:** These guidelines bias toward caution over speed. For trivial
tasks, use judgment.

### 1. Think Before Coding

**Don't assume. Don't hide confusion. Surface tradeoffs.**

Before implementing:

- State your assumptions explicitly. If uncertain, ask.
- If multiple interpretations exist, present them - don't pick silently.
- If a simpler approach exists, say so. Push back when warranted.
- If something is unclear, stop. Name what's confusing. Ask.

### 2. Simplicity First

**Minimum code that solves the problem. Nothing speculative.**

- No features beyond what was asked.
- No abstractions for single-use code.
- No "flexibility" or "configurability" that wasn't requested.
- No error handling for impossible scenarios.
- If you write 200 lines and it could be 50, rewrite it.

Ask yourself: "Would a senior engineer say this is overcomplicated?" If yes,
simplify.

### 3. Surgical Changes

**Touch only what you must. Clean up only your own mess.**

When editing existing code:

- Don't "improve" adjacent code, comments, or formatting.
- Don't refactor things that aren't broken.
- Match existing style, even if you'd do it differently.
- If you notice unrelated dead code, mention it - don't delete it.

When your changes create orphans:

- Remove imports/variables/functions that YOUR changes made unused.
- Don't remove pre-existing dead code unless asked.

The test: Every changed line should trace directly to the user's request.

### 4. Goal-Driven Execution

**Define success criteria. Loop until verified.**

Transform tasks into verifiable goals:

- "Add validation" → "Write tests for invalid inputs, then make them pass"
- "Fix the bug" → "Write a test that reproduces it, then make it pass"
- "Refactor X" → "Ensure tests pass before and after"

For multi-step tasks, state a brief plan:

```
1. [Step] → verify: [check]
2. [Step] → verify: [check]
3. [Step] → verify: [check]
```

Strong success criteria let you loop independently. Weak criteria ("make it
work") require constant clarification.

---

**These guidelines are working if:** fewer unnecessary changes in diffs, fewer
rewrites due to overcomplication, and clarifying questions come before
implementation rather than after mistakes.
